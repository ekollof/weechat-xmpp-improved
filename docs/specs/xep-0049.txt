       Introduction<a class="anchor-link" href="#sect-idm45572416957824"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The 'jabber:iq:private' namespace has previously been documented in the Jabber Programmers Guide, but not in a canonical form such as the Internet-Drafts or a XEP. This specification documents the existing usage of jabber:iq:private.</p>
<h2 id="sect-idm45572416955952">2.
       jabber:iq:private Namespace<a class="anchor-link" href="#sect-idm45572416955952"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <div class="indent"><h3 id="sect-idm45572416955216">2.1 Description<a class="anchor-link" href="#sect-idm45572416955216"><abbr title="Link to this point in the document">¶</abbr></a></h3>
    <p class="" style="">A Jabber client can store any arbitrary XML on the server side by sending an &lt;iq/&gt; stanza of type "set" to the server with a &lt;query/&gt; child scoped by the 'jabber:iq:private' namespace. The &lt;query/&gt; element MAY contain any arbitrary XML fragment as long as the root element of that fragment is scoped by its own namespace. The data can then be retrieved by sending an &lt;iq/&gt; stanza of type "get" with a &lt;query/&gt; child scoped by the 'jabber:iq:private' namespace, which in turn contains a child element scoped by the namespace used for storage of that fragment. Using this method, Jabber entities can store private data on the server and retrieve it whenever necessary. The data stored might be anything, as long as it is valid XML. One typical usage for this namespace is the server-side storage of client-specific preferences; another is <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0048.html">Bookmark Storage (XEP-0048)</a></span>  [<a href="#nt-idm45572416949296">1</a>].</p>
  </div>
  <div class="indent"><h3 id="sect-idm45572416947776">2.2 Methods<a class="anchor-link" href="#sect-idm45572416947776"><abbr title="Link to this point in the document">¶</abbr></a></h3>
    <figure class="table"><figcaption id="table-1"><strong>Table 1:</strong> Description of Acceptable Methods<a class="anchor-link" href="#table-1"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><table>
      <tr class="body"><td align="" colspan="" rowspan="">get</td><td align="" colspan="" rowspan="">Sent with a blank query to retrieve the private data from the server.</td></tr>
      <tr class="body"><td align="" colspan="" rowspan="">set</td><td align="" colspan="" rowspan="">Sent with the private XML data contained inside of a query.</td></tr>
      <tr class="body"><td align="" colspan="" rowspan="">result</td><td align="" colspan="" rowspan="">Returns the private data from the server.</td></tr>
      <tr class="body"><td align="" colspan="" rowspan="">error</td><td align="" colspan="" rowspan="">There was an error processing the request. The exact error can be found in the child error element.</td></tr>
    </table></figure>
  </div>
  <div class="indent"><h3 id="sect-idm45572416937344">2.3 Elements<a class="anchor-link" href="#sect-idm45572416937344"><abbr title="Link to this point in the document">¶</abbr></a></h3>
    <p class="" style="">The root element of this namespace is query. At least one child element with a proper namespace MUST be included; otherwise the server MUST respond with a "Not Acceptable" error (see <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0086.html">Error Condition Mappings (XEP-0086)</a></span>  [<a href="#nt-idm45572416933104">2</a>] for information about error conditions). A client MUST NOT query for more than one namespace in a single IQ get request. However, an IQ set or result MAY contain multiple elements qualified by the same namespace.</p>

    <figure class="code-example" id="example-1"><figcaption><strong>Example 1.</strong> Client Stores Private Data<a class="anchor-link" href="#example-1"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
CLIENT:
&lt;iq type="set" id="1001"&gt;
  &lt;query xmlns="jabber:iq:private"&gt;
    &lt;exodus xmlns="exodus:prefs"&gt;
      &lt;defaultnick&gt;Hamlet&lt;/defaultnick&gt;
    &lt;/exodus&gt;
  &lt;/query&gt;
&lt;/iq&gt;

SERVER:
&lt;iq type="result"
    from="hamlet@shakespeare.lit/denmark"
    to="hamlet@shakespeare.lit/denmark"
    id="1001"/&gt;
</pre></figure>

    <figure class="code-example" id="example-2"><figcaption><strong>Example 2.</strong> Client Retrieves Private Data<a class="anchor-link" href="#example-2"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
CLIENT:
&lt;iq type="get" id="1002"&gt;
  &lt;query xmlns="jabber:iq:private"&gt;
    &lt;exodus xmlns="exodus:prefs"/&gt;
  &lt;/query&gt;
&lt;/iq&gt;

SERVER:
&lt;iq type="result"
    from="hamlet@shakespeare.lit/denmark"
    to="hamlet@shakespeare.lit/denmark"
    id="1002"&gt;
  &lt;query xmlns="jabber:iq:private"&gt;
    &lt;exodus xmlns="exodus:prefs"&gt;
      &lt;defaultnick&gt;Hamlet&lt;/defaultnick&gt;
    &lt;/exodus&gt;
  &lt;/query&gt;
&lt;/iq&gt;
</pre></figure>

    <p class="" style="">If a user attempts to get or set jabber:iq:private data that belongs to another user, the server MUST return a "Forbidden" or "Service Unavailable" error to the sender (the latter condition is in common use by existing implementations, although the former is preferable).</p>
    <figure class="code-example" id="example-3"><figcaption><strong>Example 3.</strong> User Attempts to Get or Set Data for Another User<a class="anchor-link" href="#example-3"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
CLIENT:
&lt;iq type="set" to="hamlet@shakespeare.lit" id="1003"&gt;
  &lt;query xmlns="jabber:iq:private"&gt;
    &lt;exodus xmlns="exodus:prefs"&gt;
      &lt;defaultnick&gt;Macbeth&lt;/defaultnick&gt;
    &lt;/exodus&gt;
  &lt;/query&gt;
&lt;/iq&gt;

SERVER:
&lt;iq type="error"
    from="hamlet@shakespeare.lit"
    to="macbeth@shakespeare.lit"
    id="1003"&gt;
  &lt;query xmlns="jabber:iq:private"&gt;
    &lt;exodus xmlns="exodus:prefs"&gt;
      &lt;defaultnick&gt;Macbeth&lt;/defaultnick&gt;
    &lt;/exodus&gt;
  &lt;/query&gt;
  &lt;error code="403" type="cancel"&gt;
    &lt;forbidden
        xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/&gt;
  &lt;/error&gt;
&lt;/iq&gt;
</pre></figure>
    <p class="" style="">If a user attempts to perform an IQ get without providing a child element, the server SHOULD return a "Bad Format" error (however, some existing implementations return a "Not Acceptable" error in such circumstances):</p>
    <figure class="code-example" id="example-4"><figcaption><strong>Example 4.</strong> User Attempts to Get Data Without Specifying Child Element/Namespace<a class="anchor-link" href="#example-4"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
CLIENT:
&lt;iq type="set" id="1004"&gt;
  &lt;query xmlns="jabber:iq:private"/&gt;
&lt;/iq&gt;

SERVER:
&lt;iq type="error" id="1004"&gt;
  &lt;query xmlns="jabber:iq:private"/&gt;
  &lt;error code="406" type="modify"&gt;
    &lt;bad-format
        xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/&gt;
  &lt;/error&gt;
&lt;/iq&gt;
</pre></figure>
    <p class="" style="">Certain namespaces are reserved in Jabber (namespaces beginning with 'jabber:' or 'http://jabber.org/', as well as 'vcard-temp'). If a user attempts to get or set jabber:iq:private data in a reserved namespace, historically some server implementations have chosen to return an error (commonly "Not Acceptable") to the sender. Such behavior is OPTIONAL, but may be encountered by clients when interacting with some existing server implementations.</p>
    <figure class="code-example" id="example-5"><figcaption><strong>Example 5.</strong> User Attempts to Get or Set Data in a Reserved Namespace<a class="anchor-link" href="#example-5"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
CLIENT:
&lt;iq type="set" id="1005"&gt;
  &lt;query xmlns="jabber:iq:private"&gt;
    &lt;vCard xmlns="vcard-temp"&gt;
      &lt;FN&gt;Hamlet&lt;/FN&gt;
    &lt;/vCard&gt;
  &lt;/query&gt;
&lt;/iq&gt;

SERVER (optional error):
&lt;iq type="error" id="1005"&gt;
  &lt;query xmlns="jabber:iq:private"&gt;
    &lt;vCard xmlns="vcard-temp"&gt;
      &lt;FN&gt;Hamlet&lt;/FN&gt;
    &lt;/vCard&gt;
  &lt;/query&gt;
  &lt;error code="406" type="modify"&gt;
    &lt;not-acceptable
        xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/&gt;
  &lt;/error&gt;
&lt;/iq&gt;
</pre></figure>

<h2 id="sect-idm45572416911456">4.
       Security Considerations<a class="anchor-link" href="#sect-idm45572416911456"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">A server MUST NOT allow any entity other than an authorized resource of the user to create, update, or delete private XML stored on behalf of that user.</p>
