       Introduction &amp; Motivation<a class="anchor-link" href="#intro"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The XMPP instant messaging ecosystem is a federated one. This leads to many different group chat service providers existing and interesting public group chats being spread out across them. In order to provide users with a way to find public group chats (henceforth called channels) of interest to them, there needs to be a way to execute a cross-domain search based on keywords.</p>
  <p class="" style="">The protocol in this document provides a general and extensible search for channels across different domains and service types (e.g. MUC vs. MIX). It provides meta-information right in the result set, which allows searching entities to skip additional <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0030.html">Service Discovery (XEP-0030)</a></span>  [<a href="#nt-idm45588937498240">1</a>] queries against the channels themselves.</p>
  <p class="" style="">The protocol is not only useful for cross-domain search, but also as an alternative to using a <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0030.html">Service Discovery (XEP-0030)</a></span>  [<a href="#nt-idm45588937498240">1</a>] disco#items request followed by many disco#info requests on a group chat service.</p>
<h2 id="reqs">2.
       Requirements<a class="anchor-link" href="#reqs"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The protocol:</p>
  <ul class="" style="">
    <li class="" style="">must work without state on the server side. This is to allow stateless proxies to be used for pseudonymisation or anonymisation.</li>
    <li class="" style="">must allow searching the list using a free-text keyword-based search.</li>
    <li class="" style="">must allow future extensions to the search query and the result.</li>
    <li class="" style="">must allow retrieving the entire data set (although, for clarification, an operator may choose to turn this off).</li>
    <li class="" style="">must use completely machine-readable and machine-writable data.</li>
  </ul>
<h2 id="glossary">3.
       Glossary<a class="anchor-link" href="#glossary"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <div class="indent"><dl>
    
      <dt><strong>Channel</strong></dt>
      <dd>A public group chat hosted on a Group Chat Service. This can either be a <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0045.html">Multi-User Chat (XEP-0045)</a></span>  [<a href="#nt-idm45588936295488">2</a>] room, a <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0369.html">Mediated Information eXchange (MIX) (XEP-0369)</a></span>  [<a href="#nt-idm45588936285968">3</a>] channel or something else entirely.</dd>
    
    
      <dt><strong>Group Chat Service</strong></dt>
      <dd>An entity or deployment which offers multi-user chat relay, such as by <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0045.html">Multi-User Chat (XEP-0045)</a></span>  [<a href="#nt-idm45588936295488">2</a>] or <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0369.html">Mediated Information eXchange (MIX) (XEP-0369)</a></span>  [<a href="#nt-idm45588936285968">3</a>].</dd>
    
    
      <dt><strong>Search Service</strong></dt>
      <dd>An entity which offers the service described in this specification.</dd>
    
    
      <dt><strong>Searcher</strong></dt>
      <dd>An entity which requests information from the Search Service.</dd>
    
  </dl></div>

  <div class="indent"><h3 id="usecases-search">4.2 Executing a keyword search<a class="anchor-link" href="#usecases-search"><abbr title="Link to this point in the document">¶</abbr></a></h3>
    <p class="" style="">To execute a keyword search, the Searcher MAY first request the search form from the Search Service. Alternatively, the Searcher MAY use the form specified in this document with only the fields which must be implemented by the Search Service.</p>
    <p class="" style="">After obtaining the search form, the Searcher completes the form and sends it back to the Search Service. The Search Service replies with a <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0059.html">Result Set Management (XEP-0059)</a></span>  [<a href="#nt-idm45588936265680">4</a>] paginated list of results.</p>
    <p class="" style="">The search form is a form conforming to <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0068.html">Field Standardization for Data Forms (XEP-0068)</a></span>  [<a href="#nt-idm45588936260336">5</a>].</p>

    <div class="indent"><h4 id="usecases-search-request">4.2.2 Send a search request<a class="anchor-link" href="#usecases-search-request"><abbr title="Link to this point in the document">¶</abbr></a></h4>
      <p class="" style="">To request the result list for a given search query, a Searcher submits a form with the urn:xmpp:channel-search:0:search-params <code>FORM_TYPE</code>. The Searcher MAY include a <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0059.html">Result Set Management (XEP-0059)</a></span>  [<a href="#nt-idm45588936265680">4</a>] <code>&lt;set/&gt;</code> element inside the <code>&lt;search/&gt;</code> element. In either case, the Search Service may reply with a RSM-paginated result and the Searcher MUST be able to process that.</p>
      <p class="" style="">If a Searcher composes a search request using a search form template obtained by the Search Service, it MAY omit all fields it does not know or where it does not change the value already supplied by the Search Service.</p>

      <div class="indent"><h5 id="usecases-search-request-searchinvalid">4.2.2.3 Invalid Search Terms</h5>
        <p class="" style="">If the <code>q</code> field was supplied by the Searcher and the contents of the <code>q</code> field did not yield any term suitable for search, the Search Service MUST reply with an <code>&lt;bad-request/&gt;</code> error and the <code>&lt;invalid-search-terms/&gt;</code> application defined condition. The error type MUST be <code>modify</code>.</p>
        <p class="" style="">The server SHOULD include a human-readable description of the constraints for search terms which were not met in the <code>&lt;text/&gt;</code> element of the error.</p>
        <figure class="code-example" id="example-7"><figcaption><strong>Example 7.</strong> Search Service replies with the invalid-search-terms error<a class="anchor-link" href="#example-7"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">

    &lt;/text&gt;
    &lt;invalid-search-terms xmlns='urn:xmpp:channel-search:0:error'/&gt;
  &lt;/error&gt;
&lt;/iq&gt;
</pre></figure>
      </div>
      <div class="indent"><h5 id="usecases-support-request-ratelimit">4.2.2.4 Rate Limiting</h5>
        <p class="" style="">If the Search Service can not or does (by policy) not want to process the request due to excessive amounts of requests (either by the requesting entity, their domain or any other criteria), it MUST reply with an <code>&lt;resource-constraint/&gt;</code> error with type <code>wait</code>.</p>
        <p class="" style="">The application defined error condition <code>&lt;rate-limit/&gt;</code> MUST be included. This error condition has a RECOMMENDED attribute, <code>retry-after</code>, which provides the amount of seconds after which the Searcher MAY retry the request.</p>
        <p class="" style="">The Search Service MAY include a human-readable description of the rate limit and when to retry in the <code>&lt;text/&gt;</code> element.</p>
        <figure class="code-example" id="example-8"><figcaption><strong>Example 8.</strong> Search Service replies with a rate limit notification<a class="anchor-link" href="#example-8"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">

  &lt;error type='wait'&gt;
    &lt;resource-constraint xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/&gt;
    &lt;rate-limit xmlns='urn:xmpp:channel-search:0:error' retry-after='30'/&gt;
  &lt;/error&gt;
&lt;/iq&gt;
</pre></figure>
        <p class="" style=""><strong>Note:</strong> See also the rate-limiting related business rules for Searcher entities.</p>
      </div>
      <div class="indent"><h5 id="usecases-support-request-nofulllist">4.2.2.5 Rejection of Full List Retrieval</h5>
        <p class="" style="">If the Search Service can not or does (by policy) not want to allow a Searcher to retrieve the entire database of channels, it MUST reject queries which set the <code>all</code> field to true with an error as follows:</p>
        <ul class="" style="">
          <li class="" style="">If the feature is generally disabled: <code>&lt;not-allowed/&gt;</code> with type <code>cancel</code></li>
          <li class="" style="">If the feature is not offered to the Searcher based on its identity: <code>&lt;forbidden/&gt;</code> with type <code>auth</code></li>
        </ul>
        <p class="" style="">In all cases, the application defined condition <code>&lt;full-set-retrieval-rejected/&gt;</code> MUST be included.</p>
        <p class="" style="">The Search Service MAY include a human-readable description of the restrictions around full-list retrieval.</p>
        <p class="" style="">For example, if the full set retrieval had been disabled service-wide by configuration, the Search Service would reply with the following error:</p>

&lt;iq from='search.service.example' to='client@user.example' id='id3' type='error' xml:lang='en'&gt;

      <div class="indent"><h5 id="usecases-support-request-conflictingoptions">4.2.2.6 Conflicting Field Options</h5>
        <p class="" style="">If the Searcher provides form fields which are conflicting, the Search Service MUST reply with a <code>&lt;bad-request/&gt;</code> error of type <code>modify</code>. In addition, the <code>&lt;conflicting-fields/&gt;</code> application specific condition MUST be included.</p>
        <p class="" style="">Conflicting field values are those which fundamentally cannot be used in the same query in such a way that the definition of their function is still adhered to. For example, <code>q</code> restricts the results by keywords, but <code>all</code> specifies that <em>all</em> entries are returned.</p>
        <p class="" style="">The Search Service SHOULD include a human-readable description of the conflicting fields, referencing to the <code>label</code> values of the involved fields.</p>
        <p class="" style="">The <code>&lt;conflicting-fields/&gt;</code> element MAY have one or more <code>&lt;var/&gt;</code> child elements which refer to <code>var</code> values of the submitted fields. At least one of the referenced fields must be changed in order for a follow-up query to succeed.`</p>
        <p class="" style="">For example, if the Searcher has set <code>all</code> to true and provided a query in <code>q</code>, the Search Service would reply with an error similar to the following:</p>
        <figure class="code-example" id="example-10"><figcaption><strong>Example 10.</strong> Search Service replies with a conflicting-fields error<a class="anchor-link" href="#example-10"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;iq from='search.service.example' to='client@user.example' id='id3' type='error' xml:lang='en'&gt;
  &lt;error type='cancel'&gt;
    &lt;not-allowed xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/&gt;
    &lt;text xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'&gt;
      Cannot both return all results and search by keywords.
    &lt;/text&gt;
    &lt;conflicting-fields xmlns='urn:xmpp:channel-search:0:error'&gt;
      &lt;var&gt;all&lt;/var&gt;
      &lt;var&gt;q&lt;/var&gt;
    &lt;/conflicting-fields&gt;
  &lt;/error&gt;
&lt;/iq&gt;
</pre></figure>
      </div>
      <div class="indent"><h5 id="usecases-search-request-nosearch">4.2.2.7 No Search Conditions</h5>
        <p class="" style="">If no field which would define a result set and which is understood by the Search Service is present, it MUST reply with a <code>&lt;bad-request/&gt;</code> error of type <code>cancel</code>.</p>
        <p class="" style="">In addition, the <code>&lt;no-search-conditions/&gt;</code> application defined condition MUST be included.</p>
        <figure class="code-example" id="example-11"><figcaption><strong>Example 11.</strong> Search Service replies with the no-search-conditions error<a class="anchor-link" href="#example-11"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">

  &lt;error type='modify'&gt;
    &lt;bad-request xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/&gt;
    &lt;no-search-conditions xmlns='urn:xmpp:channel-search:0:error'/&gt;
  &lt;/error&gt;
&lt;/iq&gt;
</pre></figure>
        <p class="" style="">An example of this situation would be a form where neither <code>q</code> nor <code>all</code> are given.</p>

<h2 id="rules">5.
       Business Rules<a class="anchor-link" href="#rules"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <ul class="" style="">
    <li class="" style="">When sending the form template, the Search Service MUST include all fields it supports with their respective default values.</li>
    <li class="" style="">When submitting a form to the Search Service, a Searcher MAY omit all fields it either does not understand or it has left unchanged.</li>
    <li class="" style="">When submitting a form to the Search Service, a Searcher MAY omit the <code>&lt;option/&gt;</code> elements.</li>
    <li class="" style="">When receiving a search form, the Search Service MUST ignore fields with a <code>var</code> value it does not understand.</li>
    <li class="" style="">When executing a keyword search, the service may process the keyword string in implementation-defined ways. This may include interpreting quotes and other "special" characters, removing keywords which do not fit internal criteria for suitability and others.</li>
    <li class="" style="">If the Searcher receives a <code>&lt;rate-limit/&gt;</code> error, the behaviour of the Searcher depends on the <code>retry-after</code> attribute:
      <ul class="" style="">
        <li class="" style="">If the <code>retry-after</code> attribute is present, the Searcher MUST NOT send another search request before the amount of seconds indicated in the <code>retry-after</code> attribute have elapsed. There is no guarantee that the request will be accepted at that time.</li>
        <li class="" style="">If the <code>retry-after</code> attribute is <em>not</em> present, the Searcher should wait for an implementation-defined amount of time and SHOULD back off exponentially on each subsequent <code>&lt;rate-limit/&gt;</code> error.</li>
      </ul>
    </li>
    <li class="" style="">If a search request does not yield any results, the Search Service MUST reply with a <code>&lt;result/&gt;</code> without any <code>&lt;item/&gt;</code> children in a <code>type='result'</code> IQ. Specifically, it MUST NOT reply with an <code>&lt;item-not-found/&gt;</code> error.</li>
    <li class="" style="">If the <code>all</code> field is set to true and the Search Service allows this operation, all results MUST be included in the result set (and then paginated using <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0059.html">Result Set Management (XEP-0059)</a></span>  [<a href="#nt-idm45588936265680">4</a>]).</li>
  </ul>

    <div class="indent"><h4 id="impl-resultitemformat-extensibility">6.2.1 Extensibility<a class="anchor-link" href="#impl-resultitemformat-extensibility"><abbr title="Link to this point in the document">¶</abbr></a></h4>
      <p class="" style="">Search Service implementations are free to add custom child elements to <code>&lt;item/&gt;</code> elements. Searcher implementations MUST be prepared to handle any unknown elements in <code>&lt;item/&gt;</code>, for example by ignoring them.</p>
      <p class="" style="">Additional values for the <code>&lt;anonymity-mode/&gt;</code> element may be specified by future extensions. If an implementation encounters an unknown value on this field, it is RECOMMENDED to either treat it as synonymous to <code>{urn:xmpp:channel-search:0:anonymity}none</code> or request the anonymity mode from the <code>address</code> using a protocol appropriate for the channel's service.</p>

<h2 id="design">11.
       Design Considerations<a class="anchor-link" href="#design"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Instead of rolling a custom protocol for the result items, <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0055.html">Jabber Search (XEP-0055)</a></span>  [<a href="#nt-idm45588935983344">8</a>] could have been used.</p>
  <p class="" style="">While the result format of <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0055.html">Jabber Search (XEP-0055)</a></span>  [<a href="#nt-idm45588935983344">8</a>] allows for some generality, it does so in a rather restricted way. It is limited by the data formats and types expressable in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0004.html">Data Forms (XEP-0004)</a></span>  [<a href="#nt-idm45588935973584">9</a>]. Sturctured data, beyond lists of text and JIDs, can not be represented with <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0004.html">Data Forms (XEP-0004)</a></span>  [<a href="#nt-idm45588935973584">9</a>] at all. Machine-readable data would also have to be human-readable at the same time to provide a fallback view for human users. Interationalization of such human-readable data in field values is not possible with <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0004.html">Data Forms (XEP-0004)</a></span>  [<a href="#nt-idm45588935973584">9</a>].</p>
  <p class="" style="">The advantage of entities being able to process unknown fields in a degraded manner is, principally, still present in the current proposal (although with a different kind of degration).</p>
  <p class="" style="">Given the complexity of fully and correctly processing <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0004.html">Data Forms (XEP-0004)</a></span>  [<a href="#nt-idm45588935973584">9</a>] report data, the slim benefits did, in the eyes of the authors, not outweigh the costs.</p>
