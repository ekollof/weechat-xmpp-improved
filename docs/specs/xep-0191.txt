       Introduction<a class="anchor-link" href="#intro"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style=""><span class="ref" style=""><a href="http://tools.ietf.org/html/rfc3921">RFC 3921</a></span>  [<a href="#nt-idm45616102551824">1</a>] includes an XMPP protocol extension for communications blocking, which has since been moved to <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0016.html">Privacy Lists (XEP-0016)</a></span>  [<a href="#nt-idm45616102547056">2</a>]. Unfortunately, because the privacy lists extension is quite complex, it has not been widely implemented in servers and has been implemented virtually not at all in clients. This is problematic, since the ability to block communications with selected users is an important feature for an instant messaging system (and is required by <span class="ref" style=""><a href="http://tools.ietf.org/html/rfc2779">RFC 2779</a></span>  [<a href="#nt-idm45616102537872">3</a>]). However, the full power of privacy lists is not needed in order to block communications, so this document proposes a simpler blocking protocol that meets the requirement specified in <span class="ref" style=""><a href="http://tools.ietf.org/html/rfc2779">RFC 2779</a></span>  [<a href="#nt-idm45616102537872">3</a>] and can be implemented more easily in XMPP clients and servers.</p>
<h2 id="reqs">2.
       Requirements<a class="anchor-link" href="#reqs"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The requirements for communications blocking are straightforward:</p>
  <ol start="1" class="" style="">
    <li class="" style="">A user must be able to block communications with a specific JID.</li>
    <li class="" style="">A user should be able to determine which JIDs are blocked.</li>
    <li class="" style="">A user should be able to unblock communications with a specific JID.</li>
  </ol>

  <div class="indent"><h3 id="block">3.3 User Blocks JID<a class="anchor-link" href="#block"><abbr title="Link to this point in the document">¶</abbr></a></h3>
    <p class="" style="">In order for a user to block communications with a JID, the user's client shall send an IQ-set with no 'to' address (thus handled by the user's server) containing a &lt;block/&gt; element qualified by the 'urn:xmpp:blocking' namespace, where the JID to be blocked is encapsulated as the 'jid' attribute of the &lt;item/&gt; child element:</p>
    <figure class="code-example" id="example-6"><figcaption><strong>Example 6.</strong> Block command<a class="anchor-link" href="#example-6"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">

  &lt;block xmlns='urn:xmpp:blocking'&gt;
    &lt;item jid='romeo@montague.net'/&gt;
  &lt;/block&gt;
&lt;/iq&gt;
</pre></figure>
    <p class="" style="">If the server can successfully process the block command, it MUST return an IQ-result:</p>
    <figure class="code-example" id="example-7"><figcaption><strong>Example 7.</strong> Block command is successful<a class="anchor-link" href="#example-7"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;iq type='result' id='block1'/&gt;
</pre></figure>
    <p class="" style="">The server MUST also send an IQ-set to all of the user's resources that have requested the blocklist, containing the blocked item(s):</p>
    <figure class="code-example" id="example-8"><figcaption><strong>Example 8.</strong> Block "push"<a class="anchor-link" href="#example-8"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;iq to='juliet@capulet.com/chamber' type='set' id='push1'&gt;
  &lt;block xmlns='urn:xmpp:blocking'&gt;
    &lt;item jid='romeo@montague.net'/&gt;
  &lt;/block&gt;
&lt;/iq&gt;

&lt;iq to='juliet@capulet.com/balcony' type='set' id='push2'&gt;
  &lt;block xmlns='urn:xmpp:blocking'&gt;
    &lt;item jid='romeo@montague.net'/&gt;
  &lt;/block&gt;
&lt;/iq&gt;
</pre></figure>
    <p class="" style="">If the &lt;block/&gt; element does not contain at least one &lt;item/&gt; child element, the server MUST return a &lt;bad-request/&gt; error. The &lt;block/&gt; element MAY contain more than one &lt;item/&gt; child. Other standard XMPP stanza errors also apply; see <span class="ref" style=""><a href="http://tools.ietf.org/html/rfc6120">XMPP Core</a></span>  [<a href="#nt-idm45616102502720">5</a>] and <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0086.html">Error Condition Mappings (XEP-0086)</a></span>  [<a href="#nt-idm45616102498272">6</a>].</p>
    <p class="" style="">When the user blocks communications with a JID, the user's server MUST send unavailable presence information to the JID (but only if the JID is allowed to receive presence notifications from the user in accordance with the rules defined in <span class="ref" style=""><a href="http://tools.ietf.org/html/rfc3921">RFC 3921</a></span>  [<a href="#nt-idm45616102551824">1</a>]).</p>
    <p class="" style="">Once the user has blocked communications with a JID, the user's server MUST NOT deliver any XML stanzas from the JID to the user. The block remains in force until the user subsequently unblocks commmunications with the JID (i.e., the duration of the block is potentially unlimited and applies across sessions).</p>
    <p class="" style="">If a blocked JID attempts to send a stanza to the user (i.e., an inbound stanza from the user's perspective), the user's server shall handle the stanza according to the following rules:</p>
    <ul class="" style="">
      <li class="" style="">For presence stanzas (including notifications, subscriptions, and probes), the server MUST NOT respond and MUST NOT return an error.</li>
      <li class="" style="">For message stanzas, the server SHOULD return an error, which SHOULD be &lt;service-unavailable/&gt;.</li>
      <li class="" style="">For IQ stanzas of type "get" or "set", the server MUST return an error, which SHOULD be &lt;service-unavailable/&gt;. IQ stanzas of other types MUST be silently dropped by the server.</li>
    </ul>
    <p class="" style="">If the foregoing suggestions are followed, the user will appear offline to the blocked JID.</p>
    <p class="" style="">If the user attempts to send an outbound stanza to the JID, the user's server MUST NOT route the stanza to the JID but instead MUST return a &lt;not-acceptable/&gt; error containing an application-specific error condition of &lt;blocked/&gt; qualified by the 'urn:xmpp:blocking:errors' namespace:</p>
    <figure class="code-example" id="example-9"><figcaption><strong>Example 9.</strong> Error: JID is blocked<a class="anchor-link" href="#example-9"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;message type='error' from='romeo@montague.net' to='juliet@capulet.com'&gt;
  &lt;body&gt;Can you hear me now?&lt;/body&gt;
  &lt;error type='cancel'&gt;
    &lt;not-acceptable xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/&gt;
    &lt;blocked xmlns='urn:xmpp:blocking:errors'/&gt;
  &lt;/error&gt;
&lt;/message&gt;
</pre></figure>
  </div>

<h2 id="impl">4.
       Implementation Notes<a class="anchor-link" href="#impl"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">When a server receives a block command from a user, it MAY cancel any existing presence subscriptions between the user and the blocked user and MAY send a message to the blocked user; however, it is RECOMMENDED to deploy so-called "polite blocking" instead (i.e., to not cancel the presence subscriptions or send a notification). Which approach to follow is a matter of local service policy.</p>
  <p class="" style="">A service MAY also filter blocking users out of searches performed on user directories (see, for example, <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0055.html">Jabber Search (XEP-0055)</a></span>  [<a href="#nt-idm45616102458352">7</a>]); however, that functionality is out of scope for this specification.</p>
<h2 id="privacy">5.
       Relationship to Privacy Lists<a class="anchor-link" href="#privacy"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The communications blocking protocol specified herein is intended to be a user-friendly "front end" to a subset of the functionality defined by the privacy lists protocol (<span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0016.html">Privacy Lists (XEP-0016)</a></span>  [<a href="#nt-idm45616102547056">2</a>]). If a service deploys both privacy lists and the blocking command, the service MUST use the same back-end data store for both protocols. (Note: Wherever possible, this document attempts to define a protocol that is fully consistent with <span class="ref">XEP-0016</span>; if a particular aspect of functionality is not specified herein, the relevant text in <span class="ref">XEP-0016</span> shall be taken to apply.)</p>
  <p class="" style="">When implementing both XEP-0191 and XEP-0016, a service SHOULD map the blocklist to the default privacy list, where each blocked JID is represented as a privacy list item of type "jid" and action "deny".  [<a href="#nt-idm45616102449504">8</a>] If this is done and none of the user's clients ever use the privacy lists protocol, then the blocklist will always apply. This mapping has the following implications:</p>
  <ol start="1" class="" style="">
    <li class="" style=""><p class="" style="">If all of a user's clients always use the blocking command, then the default privacy list will be equivalent to the blocklist and the default privacy list will be a kind of "virtual list" (in the sense that it is never modified directly by any of the clients).</p></li>
    <li class="" style=""><p class="" style="">If one of a user's clients uses privacy lists instead of blocklists and modifies the default privacy list by removing a blocked JID or blocking a new JID, then that change will be reflected in the blocklist.</p></li>
    <li class="" style=""><p class="" style="">If one of a user's clients uses privacy lists and does anything but block or unblock a JID, then that change will not be reflected in the blocklist (since blocklists cannot represent anything except blocked JIDs).</p></li>
    <li class="" style=""><p class="" style="">If one of a user's clients removes the default privacy list and substitutes a new list for the old list, the blocked JIDs in the new default privacy list (if any) will become the new blocklist.</p></li>
    <li class="" style=""><p class="" style="">If one of a user's clients makes active something other than the default privacy list, the user may receive communications from JIDs that are blocked in the default list.</p></li>
  </ol>
  <p class="" style="">Because of the potential for confusion between block lists and privacy lists, it is NOT RECOMMENDED for a client to request both the block list and privacy lists in the same session.</p>
  <p class="" style="">The priority of blocked (jid+deny) items in the privacy list SHOULD be such that they come first in the privacy list.</p>
<h2 id="matching">6.
       JID Matching<a class="anchor-link" href="#matching"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Matching of JIDs as specified in the 'jid' attribute of the &lt;item/&gt; element SHOULD proceed in the following order (this is consistent with <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0016.html">Privacy Lists (XEP-0016)</a></span>  [<a href="#nt-idm45616102547056">2</a>]):</p>
  <ol start="" class="" style="">
    <li class="" style="">&lt;user@domain/resource&gt; (only that resource matches)</li>
    <li class="" style="">&lt;user@domain&gt; (any resource matches)</li>
    <li class="" style="">&lt;domain/resource&gt; (only that resource matches)</li>
    <li class="" style="">&lt;domain&gt; (the domain itself matches, as does any user@domain or domain/resource)</li>
  </ol>
<h2 id="security">7.
       Security Considerations<a class="anchor-link" href="#security"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">If properly implemented, this protocol extension does not introduce any new security concerns above and beyond those defined in <span class="ref" style=""><a href="http://tools.ietf.org/html/rfc3920">RFC 3920</a></span>  [<a href="#nt-idm45616102422240">9</a>] and <span class="ref">RFC 3921</span>.</p>
<h2 id="iana">8.
