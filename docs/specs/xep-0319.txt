       Introduction<a class="anchor-link" href="#intro"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">This protocol describes a way to communicate a user's last interaction time with other XMPP entities over &lt;presence/&gt; stanzas. For the purposes of this document, user interaction here refers to a human end user interacting with her device by means of a keyboard, mouse, touch screen, and so on. Based on this information XMPP clients can display the time a contact went idle or a duration for how long a contact has been idle, thereby allowing end users to estimate the expected responsiveness of their contacts.</p>
  <p class="" style="">This protocol uses absolute timestamps formatted according to the DateTime profile of <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0082.html">XMPP Date and Time Profiles (XEP-0082)</a></span>  [<a href="#nt-idm46178558038928">1</a>], indicated as value of the 'since' attribute in the &lt;idle/&gt; element.</p>
  <p class="" style="">Experience has shown a number of issues with <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0256.html">Last Activity in Presence (XEP-0256)</a></span>  [<a href="#nt-idm46178556950304">2</a>]:</p>
  <ul class="" style="">
    <li class="" style="">The use of relative durations is too vague. It requires additional information from <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0203.html">Delayed Delivery (XEP-0203)</a></span>  [<a href="#nt-idm46178556946112">3</a>] to provide a reliable user experience.</li>
    <li class="" style="">Distinguishing between the idle and last online use cases is very difficult.</li>
    <li class="" style="">It is desirable to have idle time indiciated for &lt;presence/&gt; &lt;show/&gt; values other than "away" and "xa".</li>
  </ul>
  <p class="" style="">Updating <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0256.html">Last Activity in Presence (XEP-0256)</a></span>  [<a href="#nt-idm46178556950304">2</a>] directly would be problematic as it would break compatibility with existing implementations.  Instead a new protocol is described for handling the idle time use case; the last online use case is handled by <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0312.html">PubSub Since (XEP-0312)</a></span>  [<a href="#nt-idm46178556933648">4</a>].</p>
<h2 id="usecases">2.
       Use Cases<a class="anchor-link" href="#usecases"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">There are two main use cases for this extension, explained in more detail in the following sections.</p>
  <div class="indent"><h3 id="last-interact">2.1 Presence with Last Interaction<a class="anchor-link" href="#last-interact"><abbr title="Link to this point in the document">¶</abbr></a></h3>
  <p class="" style="">After a user has not interacted with her device for some amount of time the user wants to inform her contacts about this fact. The client sends a &lt;presence/&gt; stanza with time of last interaction.</p>
  <figure class="code-example" id="example-1"><figcaption><strong>Example 1.</strong> Time of Last User Interaction in Auto-Away<a class="anchor-link" href="#example-1"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;presence from='juliet@capulet.com/balcony'&gt;
  &lt;show&gt;away&lt;/show&gt;
  &lt;idle xmlns='urn:xmpp:idle:1' since='1969-07-21T02:56:15Z'/&gt;
&lt;/presence&gt;
</pre></figure>
  <p class="" style="">The amount of time the user has to be idle before a client sends this enhanced presence is application-specific; it is suggested that a sensible default interval of 5 minutes be used.</p>
  </div>
  <div class="indent"><h3 id="back-from-idle">2.2 Presence Indicating User Coming Back From Idle<a class="anchor-link" href="#back-from-idle"><abbr title="Link to this point in the document">¶</abbr></a></h3>
    <p class="" style="">When a user comes back and uses her device again the client informs user's contacts by sending a normal presence stanza like shown in the following example, omiting the &lt;idle/&gt; element.</p>
    <figure class="code-example" id="example-2"><figcaption><strong>Example 2.</strong> Presence Indicating Return to Device<a class="anchor-link" href="#example-2"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">

</pre></figure>
  </div>

<h2 id="security">5.
       Security Considerations<a class="anchor-link" href="#security"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The security considerations of <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0082.html">XMPP Date and Time Profiles (XEP-0082)</a></span>  [<a href="#nt-idm46178558038928">1</a>] apply to this protocol.</p>
  <p class="" style="">This specification introduces no new security or privacy concerns. While including a last user interaction notation in &lt;presence/&gt; updates can enable recipients to determine exactly when a user has stopped interacting with her XMPP client or even their system, this information is in essence already available if the user's client publishes timely presence updates.</p>
