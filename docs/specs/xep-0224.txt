       Introduction<a class="anchor-link" href="#intro"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Even though a client might be available (as stated in the most recent presence stanza), the user this client belongs to might not be focused on the client currently. <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0132.html">Presence Obtained via Kinesthetic Excitation (POKE) (XEP-0132)</a></span>  [<a href="#nt-idm45481168951712">1</a>] defines a method for a physical test of user presence. Since this requires special hardware that cannot be assumed to be available, this XEP defines a software-only implementation where no direct feedback is expected. This feature is known as 'nudge' or 'buzz' in some non-XMPP IM protocols.</p>
  <p class="" style="">It was discussed whether this feature belongs in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0085.html">Chat State Notifications (XEP-0085)</a></span>  [<a href="#nt-idm45481167763760">2</a>]. However, the semantics are inherently different, since Chat State Notifications describe the sender's state, not a request to change the receiver's. Thus, a separate extension is desirable.</p>
<h2 id="reqs">2.
       Requirements<a class="anchor-link" href="#reqs"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The specification addresses remotely getting the user's attention in a more assertive way than simple text messages.</p>
<h2 id="protocol">3.
       Protocol<a class="anchor-link" href="#protocol"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">In the following conversation, a user talks to somebody, but this user doesn't respond. The second inquiry includes an attention extension.</p>
  <figure class="code-example" id="example-1"><figcaption><strong>Example 1.</strong> User sends a regular message<a class="anchor-link" href="#example-1"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;message from='calvin@usrobots.lit/lab'
         to='herbie@usrobots.lit/home'
         type='chat'&gt;
  &lt;body&gt;All right, then, Herbie, give! We're waiting.&lt;/body&gt;
&lt;/message&gt;
</pre></figure>
  <p class="" style="">When no reply is received, the sending user might want to grab the other's attention. This is done by sending a message that includes an &lt;attention/&gt; element qualified by the 'urn:xmpp:attention:0' namespace (see <a href="#registrar-versioning">Namespace Versioning</a> regarding the possibility of incrementing the version number). Note: The message MAY include a &lt;body/&gt; element.</p>
  <figure class="code-example" id="example-2"><figcaption><strong>Example 2.</strong> User tries to capture the other's attention<a class="anchor-link" href="#example-2"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;message from='calvin@usrobots.lit/lab'
         to='herbie@usrobots.lit/home'
         type='headline'&gt;
  &lt;attention xmlns='urn:xmpp:attention:0'/&gt;
  &lt;body&gt;Why don't you answer, Herbie?&lt;/body&gt;
&lt;/message&gt;
</pre></figure>
  <p class="" style="">Finally, the receiving user notices the urgency of the message and responds.</p>
  <figure class="code-example" id="example-3"><figcaption><strong>Example 3.</strong> The user whose attention has been captured responds.<a class="anchor-link" href="#example-3"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;message from='herbie@usrobots.lit/home'
         to='calvin@usrobots.lit/lab'
         type='chat'&gt;
  &lt;body&gt;I cannot. You know I cannot! Dr. Bogert and Dr. Lanning don't want me to.&lt;/body&gt;
&lt;/message&gt;
</pre></figure>
<h2 id="rules">4.
       Business Rules<a class="anchor-link" href="#rules"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The following rules apply to generating and processing of the attention extension.</p>
  <ol start="" class="" style="">
    <li class="" style="">Before sending an attention message stanza, the client SHOULD confirm support for it in the other client as described under <a href="#disco">Determining Support</a>.</li>
    <li class="" style="">The message stanza containing the attention extension MAY contain a body and/or other extensions, which is to be displayed along with executing the attention event.</li>
    <li class="" style="">In message stanzas containing either <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0203.html">Delayed Delivery (XEP-0203)</a></span>  [<a href="#nt-idm45481167742592">3</a>] data, attention extensions MUST be ignored, since the attention request is an instant event which SHOULD NOT be replayed after a delay.</li>
    <li class="" style="">Messages containing an attention extension SHOULD use the headline message type to avoid offline storage.</li>
    <li class="" style="">The attention extension MUST NOT be sent in &lt;iq/&gt; stanzas, since use of this feature is part of a messaging conversation.</li>
  </ol>
<h2 id="disco">5.
       Determining Support<a class="anchor-link" href="#disco"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">If an entity wishes to receive the attention extension, it MUST advertise that fact in its responses to <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0030.html">Service Discovery (XEP-0030)</a></span>  [<a href="#nt-idm45481167734464">4</a>] information ("disco#info") requests by returning a feature of "urn:xmpp:attention:0":</p>
  <figure class="code-example" id="example-4"><figcaption><strong>Example 4.</strong> A disco#info query<a class="anchor-link" href="#example-4"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;iq type='get'
    from='calvin@usrobots.lit/lab'
    to='herbie@usrobots.lit/home'
    id='disco1'&gt;
  &lt;query xmlns='http://jabber.org/protocol/disco#info'/&gt;
&lt;/iq&gt;
</pre></figure>
  <figure class="code-example" id="example-5"><figcaption><strong>Example 5.</strong> A disco#info response<a class="anchor-link" href="#example-5"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;iq type='result'
    from='herbie@usrobots.lit/home'
    to='calvin@usrobots.lit/lab'
    id='disco1'&gt;
  &lt;query xmlns='http://jabber.org/protocol/disco#info'&gt;
    &lt;feature var='urn:xmpp:attention:0'/&gt;
  &lt;/query&gt;
&lt;/iq&gt;
</pre></figure>
  <p class="" style="">In order for an application to determine whether an entity supports this protocol, where possible it SHOULD use the dynamic, presence-based profile of service discovery defined in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0115.html">Entity Capabilities (XEP-0115)</a></span>  [<a href="#nt-idm45481167725760">5</a>]. However, if an application has not received entity capabilities information from an entity, it SHOULD use explicit service discovery instead.</p>
<h2 id="impl">6.
       Implementation Notes<a class="anchor-link" href="#impl"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The implementation of the alert is up to the developer. Possible behavior includes:</p>
  <ul class="" style="">
    <li class="" style="">Shaking the window.</li>
    <li class="" style="">Playing a specific sound not used for any other event.</li>
    <li class="" style="">Flashing the screen.</li>
    <li class="" style="">Enabling external hardware such as flashing lights.</li>
    <li class="" style="">Let it be user customizable.</li>
  </ul>
  <p class="" style="">Because some users might not want this feature to disturb them, a client MUST either (1) allow the user to disable support or (2) disable the feature by default and process attention requests only if the user has explicitly enabled support. When the feature is disabled, it MUST NOT be advertised in disco#info.</p>
  <p class="" style="">Rate-limiting might be desirable in some implementations.</p>
  <p class="" style="">Formal feedback in response to the attention request to the requesting user is not specified, and so the request might be silently dropped.</p>
<h2 id="security">7.
       Security Considerations<a class="anchor-link" href="#security"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">It is RECOMMENDED that a client accept message stanzas containing the attention extension only contacts that are in the user's roster or with whom the user's client is currently sharing directed presence, mainly to prevent the user from being annoyed by attention requests from random entities on the network. A client could implement finer-grained control if desired (e.g., allow attention requests only from entities in a particular roster group).</p>
