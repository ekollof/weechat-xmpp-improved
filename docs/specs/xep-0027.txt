       Introduction<a class="anchor-link" href="#intro"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The Jabber community has long acknowledged the need for privacy and security features in a well-rounded instant messaging system.  Unfortunately, finding a consensus solution to the problem of end-to-end encryption during the community's younger days was not easy.  Eventually, early contributors created a quick solution using OpenPGP (<span class="ref" style=""><a href="http://tools.ietf.org/html/rfc4880">RFC 4880</a></span>  [<a href="#nt-idm46101443377104">1</a>]).  This specification documents the OpenPGP solution as it is used today, so that others may interoperate with clients that support it.  This document is not intended to present a standard, because more complete solutions are being investigated.</p>
  <p class="" style="">All operations described here are done with standard OpenPGP software such as <a href="http://www.gnupg.org/">GnuPG</a>.  All program output is US-ASCII armored output with the headers removed.  This allows for easy transportation of the program output directly in the XML.  All keys are exchanged using OpenPGP key servers, and usually are retrieved when a signed &lt;presence/&gt; stanza is received (key retrieval does not happen in-band).</p>
<h2 id="signing">2.
       Signing<a class="anchor-link" href="#signing"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Signing enables a sender to verify that they sent a certain block of text.  In Jabber, signing uses the 'jabber:x:signed' namespace, and is primarily used with &lt;presence/&gt;, but may also be used with &lt;message/&gt;.  The text that is signed MAY be the empty string. When signing presence, the sender SHOULD sign the XML character data of the &lt;status&gt; element.  The sender SHOULD sign presence using the private key whose KeyID corresponds to the public key to be used in encrypting messages (see below).</p>

<h2 id="encrypting">3.
       Encrypting<a class="anchor-link" href="#encrypting"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Encryption enables the sender to encrypt a message to a specific recipient.  This is accomplished using the 'jabber:x:encrypted' namespace in conjunction with &lt;message/&gt; stanzas.  Because a block of text is necessary in order to have something to encrypt, &lt;message/&gt; stanzas intended to be encrypted have the same restrictions as signing (see above).  The data encrypted MUST be the XML character data of the &lt;body&gt; element.  The sender SHOULD encrypt the message body using the public key whose KeyID corresponds to the private key used in signing presence (see above).</p>

  <p class="" style="">It is considered polite to include an unencrypted message &lt;body/&gt; explaining that the actual message body is encrypted.  This helps if the client experiences an error while decrypting the message, or if the user's a client that does not support encryption (although generally this should not happen, since the signed presence can be used to indicate that a client accepts encrypted messages).</p>
<h2 id="security">4.
       Security Considerations<a class="anchor-link" href="#security"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The method defined herein has the following security issues:</p>
  <ul class="" style="">
    <li class="" style="">Key exchange relies on the web of trust model used on the OpenPGP keys network.</li>
    <li class="" style="">There is no mechanism for checking a fingerprint or ownership of a key other than checking the user IDs on a key.</li>
    <li class="" style="">When the recipient is not mentioned in the encrypted body, replay attacks are possible on messages.</li>
    <li class="" style="">Replay of the signed &lt;presence/&gt; status is possible.</li>
    <li class="" style="">It relies on signing or encryption of XML character data; therefore, it does not support signing or encryption of &lt;iq/&gt; stanzas, and it allows signing of the presence &lt;status/&gt; element and encryption of the message &lt;body/&gt; element only. Thus the method is not acceptable when signing or encryption of full stanzas is required.</li>
    <li class="" style="">It does not enable both signing and encryption of a stanza, only signing of the presence status and encryption of the message body.</li>
  </ul>
<h2 id="issues">5.
       Other Known Issues<a class="anchor-link" href="#issues"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">In addition to the security considerations listed above, there are several other known issues with this method:</p>
  <ul class="" style="">
    <li class="" style="">It is limited to PGP keys and does not support X.509 certificates, Kerberos, RSA keys, etc.</li>
    <li class="" style="">It does not include feature negotiation; instead, signed &lt;presence/&gt; is used as an indicator of support.  Because of the lack of negotiation it is possible for encrypted &lt;message/&gt; elements to be stored offline and then read by a client that cannot support them.</li>
    <li class="" style="">It is verbose (the example encrypted &lt;message/&gt; is "Hi").</li>
  </ul>
