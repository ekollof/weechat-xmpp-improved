       Introduction<a class="anchor-link" href="#intro"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">This document defines an extension mechanism for capturing "extended presence" data about user activities, above and beyond availability as defined in <span class="ref" style=""><a href="http://tools.ietf.org/html/rfc6121">XMPP IM</a></span>  [<a href="#nt-idm45588254371536">1</a>] (e.g., the 'away', 'extended away', and 'dnd' values of the &lt;show/&gt; child of the &lt;presence/&gt; stanza).</p>
<h2 id="proto">2.
       Protocol<a class="anchor-link" href="#proto"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <div class="indent"><h3 id="proto-format">2.1 Data Format<a class="anchor-link" href="#proto-format"><abbr title="Link to this point in the document">¶</abbr></a></h3>
    <p class="" style="">Information about user activities is provided by the user and propagated on the network by the user's client. The information is structured by means of an &lt;activity/&gt; element that is qualified by the 'http://jabber.org/protocol/activity' namespace. The general activity is provided as the element name of a first-level child of the &lt;activity/&gt; element (e.g., &lt;relaxing/&gt;); one such general activity element is REQUIRED. The general activity element MAY contain a child element that specifies a more particular form of the general activity (e.g., &lt;partying/&gt;). The user MAY also specify a natural-language description of the activity in the OPTIONAL &lt;text/&gt; child of the &lt;activity/&gt; element. Here is an example:</p>
    <figure class="code"><figcaption></figcaption><pre class="prettyprint">
&lt;activity xmlns='http://jabber.org/protocol/activity'&gt;
  &lt;relaxing&gt;
    &lt;partying/&gt;
  &lt;/relaxing&gt;
  &lt;text xml:lang='en'&gt;My nurse&amp;apos;s birthday!&lt;/text&gt;
&lt;/activity&gt;
</pre></figure>
    <p class="" style="">Instead of (but not in addition to) one of the specific activity elements defined herein, an application MAY include a properly-namespaced child element for the specific activity. Here is an example:</p>
    <figure class="code"><figcaption></figcaption><pre class="prettyprint">
&lt;activity xmlns='http://jabber.org/protocol/activity'&gt;
  &lt;relaxing&gt;
    &lt;tanning xmlns='http://www.ilovetanning.info'/&gt;
  &lt;/relaxing&gt;
&lt;/activity&gt;
</pre></figure>
    <p class="" style="">Finally, one of the specific activity elements defined herein MAY itself contain a properly-namespaced child element that provides more detailed information about the specific activity. Here is an example:</p>
    <figure class="code"><figcaption></figcaption><pre class="prettyprint">
&lt;activity xmlns='http://jabber.org/protocol/activity'&gt;
  &lt;inactive&gt;
    &lt;sleeping&gt;
      &lt;hibernating xmlns='http://www.ursus.info/states'/&gt;
    &lt;/sleeping&gt;
  &lt;/inactive&gt;
&lt;/activity&gt;
</pre></figure>
    <p class="" style="">In accordance with <span class="ref" style=""><a href="http://tools.ietf.org/html/rfc6120">XMPP Core</a></span>  [<a href="#nt-idm45588254357456">2</a>], the receiving application MUST ignore a specific activity element or detailed activity element if it does not understand the namespace that qualifies the element.</p>
  </div>
  <div class="indent"><h3 id="proto-pubsub">2.2 Pubsub Transport<a class="anchor-link" href="#proto-pubsub"><abbr title="Link to this point in the document">¶</abbr></a></h3>
    <p class="" style="">Activity information SHOULD be communicated and transported by means of the <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0060.html">Publish-Subscribe (XEP-0060)</a></span>  [<a href="#nt-idm45588254352576">3</a>] subset specified in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0163.html">Personal Eventing Protocol (XEP-0163)</a></span>  [<a href="#nt-idm45588254345808">4</a>]. Because activity information is not pure presence information and can change independently of the user's availability, it SHOULD NOT be provided as an extension to &lt;presence/&gt;.</p>
  <figure class="code-example" id="example-1"><figcaption><strong>Example 1.</strong> User Publishes Activity<a class="anchor-link" href="#example-1"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;iq type='set'
    from='juliet@capulet.lit/ca486eba-0f9a-11dc-8835-000bcd821bfb'
    id='publish1'&gt;
  &lt;pubsub xmlns='http://jabber.org/protocol/pubsub'&gt;
    &lt;publish node='http://jabber.org/protocol/activity'&gt;
      &lt;item&gt;
        &lt;activity xmlns='http://jabber.org/protocol/activity'&gt;
          &lt;relaxing&gt;
            &lt;partying/&gt;
          &lt;/relaxing&gt;
          &lt;text xml:lang='en'&gt;My nurse&amp;apos;s birthday!&lt;/text&gt;
        &lt;/activity&gt;
      &lt;/item&gt;
    &lt;/publish&gt;
  &lt;/pubsub&gt;
&lt;/iq&gt;
</pre></figure>
    <p class="" style="">The activity is then delivered to all subscribers:</p>
    <figure class="code-example" id="example-2"><figcaption><strong>Example 2.</strong> Activity is Delivered to All Subscribers<a class="anchor-link" href="#example-2"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;message
    from='juliet@capulet.lit'
    to='romeo@montague.lit'&gt;
  &lt;event xmlns='http://jabber.org/protocol/pubsub#event'&gt;
    &lt;items node='http://jabber.org/protocol/activity'&gt;
      &lt;item id='b5ac48d0-0f9c-11dc-8754-001143d5d5db'&gt;
        &lt;activity xmlns='http://jabber.org/protocol/activity'&gt;
          &lt;relaxing&gt;
            &lt;partying/&gt;
          &lt;/relaxing&gt;
          &lt;text xml:lang='en'&gt;My nurse&amp;apos;s birthday!&lt;/text&gt;
        &lt;/activity&gt;
      &lt;/item&gt;
    &lt;/items&gt;
  &lt;/event&gt;
&lt;/message&gt;
</pre></figure>
    <p class="" style="">In order to indicate that the user is no longer publishing activities, the user's client shall send an empty &lt;activity/&gt; element, which can be considered a "stop command" for user activities:</p>
    <figure class="code-example" id="example-3"><figcaption><strong>Example 3.</strong> User Disables Publishing<a class="anchor-link" href="#example-3"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;iq from='juliet@capulet.lit/balcony'
    id='publish1'
    type='set'&gt;
  &lt;pubsub xmlns='http://jabber.org/protocol/pubsub'&gt;
    &lt;publish node='http://jabber.org/protocol/activity'&gt;
      &lt;item&gt;
        &lt;activity xmlns='http://jabber.org/protocol/activity'/&gt;
      &lt;/item&gt;
    &lt;/publish&gt;
  &lt;/pubsub&gt;
&lt;/iq&gt;
</pre></figure>
    <figure class="code-example" id="example-4"><figcaption><strong>Example 4.</strong> Empty Activity Information is Delivered to All Subscribers<a class="anchor-link" href="#example-4"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><pre class="prettyprint">
&lt;message
    from='juliet@capulet.lit'
    to='romeo@montague.net'&gt;
  &lt;event xmlns='http://jabber.org/protocol/pubsub#event'&gt;
    &lt;items node='http://jabber.org/protocol/activity'&gt;
      &lt;item id='b5ac48d0-0f9c-11dc-8754-001143d5d5db'&gt;
        &lt;activity xmlns='http://jabber.org/protocol/activity'/&gt;
      &lt;/item&gt;
    &lt;/items&gt;
  &lt;/event&gt;
&lt;/message&gt;
</pre></figure>
  </div>
<h2 id="activities">3.
       Activity Values<a class="anchor-link" href="#activities"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Each activity has a REQUIRED general category and an OPTIONAL specific instance. One can understand each specifier as '[user] is [activity]' (e.g., 'Juliet is partying'), where the relevant value is the most specific activity provided (e.g., specifically "partying" rather than generally "relaxing").</p>
  <p class="" style="">The activity values defined in this taxonomy are as follows, where the first indentation level is the general category and the second indentation level is the specific instance. Note: The specific activity elements are RECOMMENDED as forms of the general activities shown below, but can be included under any general activity (e.g., "gardening" could be used as the specific activity under "relaxing" rather than "doing_chores").</p>

<h2 id="i18n">5.
       Internationalization Considerations<a class="anchor-link" href="#i18n"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The XML character data values of the &lt;text/&gt; element are intended for presentation to human users; therefore, if a &lt;text/&gt; element is included the sending application SHOULD also ensure that the &lt;text/&gt; element or the parent &lt;activity/&gt; element possesses an 'xml:lang' attribute with an appropriate value.</p>
<h2 id="security">6.
       Security Considerations<a class="anchor-link" href="#security"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Because user activities may be published to a large number of pubsub subscribers, users should take care in approving subscribers and in characterizing their current activities.</p>
