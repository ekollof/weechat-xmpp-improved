href="#appendix-biblatex">Bib(La)Tex Entry</a></li></ol></nav><h2 id="intro">1.
       Introduction<a class="anchor-link" href="#intro"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Although the XMPP protocol extension defined in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0090.html">Legacy Entity Time (XEP-0090)</a></span>  [<a href="#nt-idm45052960763584">1</a>] provides a way to discover the time at another entity, it has several limitations:</p>
  <ul class="" style="">
    <li class="" style=""><p class="" style="">The 'jabber:iq:time' namespace specified in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0090.html">Legacy Entity Time (XEP-0090)</a></span>  [<a href="#nt-idm45052960763584">1</a>] requires communication of time only in UTC. While this is useful for UTC synchronization (e.g., if a client wants to synchronize with its server), it does not enable one entity to know the other entity's offset from UTC.</p></li>
    <li class="" style=""><p class="" style="">The timezone may be specified in a natural language (English) name via the &lt;tz/&gt; element, but not in a numeric offest. The name may be not understood by the requesting entity since there is no reliable and canonical list of timezone names  [<a href="#nt-idm45052960755024">2</a>] and in practice the XML character data of the &lt;tx/&gt; element is effectively useless.</p></li>
    <li class="" style=""><p class="" style="">The responding entity may provide a user-friendly datetime format via the &lt;display/&gt; element, but this too is effectively useless since datetime formats vary widely by culture and nation.</p></li>
    <li class="" style=""><p class="" style="">The 'jabber:iq:time' namespace specified in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0090.html">Legacy Entity Time (XEP-0090)</a></span>  [<a href="#nt-idm45052960763584">1</a>] (first developed in 1999 or 2000) is not consistent with the recommended date and time profiles for XMPP protocols defined in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0082.html">XMPP Date and Time Profiles (XEP-0082)</a></span>  [<a href="#nt-idm45052960744624">3</a>] (written in 2003).</p></li>
  </ul>
  <p class="" style="">To overcome these limitations, this document defines a replacement for <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0090.html">Legacy Entity Time (XEP-0090)</a></span>  [<a href="#nt-idm45052960763584">1</a>] which enables communication of an entity's UTC time and numeric time zone offset while adhering to <span class="ref">XEP-0082</span>.</p>
<h2 id="protocol">2.
       Protocol Definition<a class="anchor-link" href="#protocol"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">The protocol defined herein provides a standard way for XMPP entities to exchange information about the local time. The information is communicated in a request/response pair using an &lt;iq/&gt; element that contains a &lt;time/&gt; element qualified by the 'urn:xmpp:time' namespace. The following children of the &lt;time/&gt; element are defined for use in IQ stanzas of type 'result':</p>
  <figure class="table"><figcaption id="table-1"><strong>Table 1:</strong> Child Elements<a class="anchor-link" href="#table-1"><abbr title="Link to this point in the document">¶</abbr></a></figcaption><table>
    <tr class="body">
      <th colspan="" rowspan="">Element</th>
      <th colspan="" rowspan="">Definition</th>
      <th colspan="" rowspan="">Inclusion</th>
    </tr>
    <tr class="body">
      <td align="" colspan="" rowspan="">&lt;tzo/&gt;</td>
      <td align="" colspan="" rowspan="">The entity's numeric time zone offset from UTC. The format MUST conform to the Time Zone Definition (TZD) specified in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0082.html">XMPP Date and Time Profiles (XEP-0082)</a></span>  [<a href="#nt-idm45052960744624">3</a>].</td>
      <td align="" colspan="" rowspan="">REQUIRED</td>
    </tr>
    <tr class="body">
      <td align="" colspan="" rowspan="">&lt;utc/&gt;</td>
      <td align="" colspan="" rowspan="">The UTC time according to the responding entity. The format MUST conform to the dateTime profile specified in <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0082.html">XMPP Date and Time Profiles (XEP-0082)</a></span>  [<a href="#nt-idm45052960744624">3</a>] and MUST be expressed in UTC.</td>
      <td align="" colspan="" rowspan="">REQUIRED</td>
    </tr>
  </table></figure>

<h2 id="impl">5.
       Implementation Notes<a class="anchor-link" href="#impl"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">This protocol was designed in a way that makes migration from <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0090.html">Legacy Entity Time (XEP-0090)</a></span>  [<a href="#nt-idm45052960763584">1</a>] straightforward. This document specifies a different format for the XML character data of the &lt;utc&gt; element (compliant with <span class="ref">XEP-0082</span>) and specifies a new &lt;tzo&gt; element for the numeric offset from UTC, while removing the formerly optional and effectively useless &lt;display/&gt; and &lt;tz/&gt; elements.</p>
  <p class="" style="">Implementations that support <span class="ref" style=""><a href="https://xmpp.org/extensions/xep-0090.html">Legacy Entity Time (XEP-0090)</a></span>  [<a href="#nt-idm45052960763584">1</a>] should support the protocol defined herein as soon as possible, but should continue to support the protocol defined in <span class="ref">XEP-0090</span> for backwards compatibility until the status of that specification is changed to Obsolete.</p>
<h2 id="security">6.
       Security Considerations<a class="anchor-link" href="#security"><abbr title="Link to this point in the document">¶</abbr></a></h2>
  <p class="" style="">Revealing an entity's numeric time zone offset may leak limited information about the entity's current location. If the entity's understanding of UTC is far off from actual UTC, revealing that discrepancy may make it possible for an attacker to send XML stanzas that appear to be in the past or future even though they are not; therefore an entity should use the Network Time Protocol (<span class="ref" style=""><a href="http://tools.ietf.org/html/rfc0958">RFC 958</a></span>  [<a href="#nt-idm45052960688576">6</a>]) or a similar technology to stay synchronized with actual UTC.</p>
